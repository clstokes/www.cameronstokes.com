Cross-Origin Resource Sharing and Apache Httpd

Dec 26th, 2010

After asking Amazon to support the W3C Cross-Origin Resource Sharing specification for the Route 53 APIs I decided to look into what it would take to implement the necessary headers in the Apache web server.

The specification describes several headers that need to be set, but we’ll only focus on the main header, Access-Control-Allow-Origin. Setting the header is easy; determining when to set it and what to set it to is the trickier part. After a bit of digging through the HTTP Server documentation I found the necessary configuration. Using mod_headers and mod_setenvif you can easily support dynamic headers to support cross-origin resource sharing.

Here’s the config:

Line 1 uses SetEnvIf to set an environment variable if the request attribute matches the regular expression. In this case we’re matching on the Origin header for example.com or example.org and if it matches, the AccessControlAllowOrigin environment variable is set to the value of the Origin header as specified by the $1 backreference.

Line 2 adds the Access-Control-Allow-Origin header to the response. The value is set to the AccessControlAllowOrigin environment variable from line 1. The env=AccessControlAllowOrigin directive means the header will only be added if the AccessControlAllowOrigin environment variable has been set.

The specification states you can support a wildcard value (*) for the Access-Control-Allow-Origin header or a comma-delimited list of domains, but with this configuration you can be specific in permitting access to individual domains without specifying everyone you’ve given access to. The HTTP access control documentation at the Mozilla Developer Network provides a great reference for how to use the cross-origin resource sharing functionality on the server and client-side.
